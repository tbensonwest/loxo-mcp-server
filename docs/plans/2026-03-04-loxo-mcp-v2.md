# Loxo MCP Server v2 Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Add 6 new tools (candidate create/update, candidate brief, activity history, job pipeline, apply to job), merge 3 open PRs (2 as-is, 1 partial), and improve search tool descriptions.

**Architecture:** All changes are in `src/index.ts`. Each new tool follows the existing pattern: define in `ListToolsRequestSchema` handler, implement in `CallToolRequestSchema` switch. The `makeRequest` helper handles auth and error formatting — use it for all API calls. POST/PATCH use `application/x-www-form-urlencoded` with bracket notation (`person[name]`). The composite `loxo_get_candidate_brief` uses `Promise.all` for parallel fetches.

**Tech Stack:** TypeScript, `@modelcontextprotocol/sdk`, `zod`, Node.js `fetch`. No test framework — verification is `npm run build` (TypeScript compilation) after each task.

  const { id } = args as any;

  const [profile, emails, phones, activitiesResponse] = await Promise.all([
    makeRequest<Candidate>(`/${env.LOXO_AGENCY_SLUG}/people/${id}`),
    makeRequest<EmailInfo[]>(`/${env.LOXO_AGENCY_SLUG}/people/${id}/emails`),

    || activitiesResponse?.events
    || activitiesResponse
    || [];

  const brief = {
    profile,
    emails,
    phones,
    recent_activities: Array.isArray(recentActivities)
      ? recentActivities.slice(0, 5)
      : [],
  };

  const { text } = truncateResponse(JSON.stringify(brief, null, 2));
  return { content: [{ type: "text", text }] };

## Notes for the Implementer

**API field names to watch:** Loxo's request body uses bracket notation (`person[name]`, not `name`). The OAS doc in `.claudedocs/oas-analysis-and-implementation-plan.md` has verified endpoint paths. If an endpoint returns a 404 or 422, check `.claudedocs/oas-analysis-and-implementation-plan.md` for the correct path.

**Response field names:** Loxo's JSON responses use snake_case. The API may wrap results in a key like `person_events`, `job_contacts`, etc. — the handlers above check multiple possible keys with fallbacks.

**The composite tool:** `loxo_get_candidate_brief` uses `Promise.all` — if any sub-request fails (e.g. no emails on file returns 404), the whole tool fails. If this becomes an issue, wrap each inner call in a try/catch that returns an empty array on error.

**No test framework:** Verification is `npm run build`. TypeScript is strict — compilation success means the code is structurally correct. Functional testing requires real Loxo credentials.
